# YAMLconfig

YAMLconfig is an authenticated Django web application for building, previewing,
and exporting Docker configuration without writing every file by hand.

The current application supports:

- Docker Compose YAML generation
- Dockerfile generation
- Per-user configuration projects
- Structured services, networks, named volumes, and build options
- Live syntax-highlighted output previews
- Compose, Dockerfile, and ZIP downloads
- Account signup, mandatory email verification, login, logout, password
  management, and email management through django-allauth

Ansible playbooks, GitHub Actions workflows, Kubernetes manifests, and other
configuration formats are possible future directions; they are not implemented
in the current version.

## How it works

YAMLconfig stores configuration as structured database records rather than as
editable YAML text. A project owns related services, networks, named volumes,
and scoped options. The generator reads those records and produces either a
Docker Compose document or a Dockerfile.

```text
Browser
  → Django URL and authenticated view
  → Model form validation
  → SQLite project records
  → generator/yaml_builder.py
  → live preview or file download
```

Generated output is derived data. It is rebuilt from the current database state
for every preview and download request and is not cached in the database. This
means changes made through the normal UI, Django admin, or direct ORM operations
are visible the next time output is requested.

Every normal project-facing view is login-protected. Project lookups are scoped
to the current owner, so one user cannot view, modify, or download another
user's project.

## Requirements

- Python 3.10 or newer
- `pip`
- A virtual environment is strongly recommended
- Internet access in the browser for the current Tailwind, Alpine.js, and
  CodeMirror CDN assets

Docker is not required to run YAMLconfig itself. Docker is only required if you
want to build or run the files generated by the application.

## Local setup

Run these commands from the repository root.

### 1. Create and activate a virtual environment

Linux or macOS:

```bash
python3 -m venv venv
source venv/bin/activate
```

Windows PowerShell:

```powershell
py -m venv venv
.\venv\Scripts\Activate.ps1
```

### 2. Install dependencies

```bash
python -m pip install --upgrade pip
python -m pip install -r requirements.txt
```

### 3. Apply database migrations

```bash
python manage.py migrate
```

This creates the SQLite database and installs the Django, allauth, axes, and
generator tables.

### 4. Start the development server

```bash
python manage.py runserver
```

Open:

```text
http://127.0.0.1:8000/
```

The dashboard redirects anonymous users to the allauth login page. Use the
signup link to create an account.

### 5. Verify the development account

Email verification is mandatory. With the default development settings, Django
prints the verification email and confirmation URL in the terminal running the
server. Open that URL in the same browser to verify the address and complete
login.

### Optional: create an administrator

```bash
python manage.py createsuperuser
```

The Django admin is then available at:

```text
http://127.0.0.1:8000/admin/
```

## Environment configuration

Settings are read with `os.getenv()`. A `.env` file is not loaded
automatically—`python-dotenv` and similar loaders are not part of this project.
Export variables in the shell, configure them in your process manager, or
explicitly source a shell-compatible file before starting Django.

Example for Bash:

```bash
export DJANGO_SECRET_KEY="replace-this-with-a-long-random-value"
export DJANGO_DEBUG="true"
export DJANGO_ALLOWED_HOSTS="127.0.0.1,localhost"
python manage.py runserver
```

Recognized variables:

| Variable | Development default | Purpose |
| --- | --- | --- |
| `DJANGO_SECRET_KEY` | `django-secret-key` | Django signing and cryptographic secret |
| `DJANGO_DEBUG` | `true` | Enables Django debug mode for `1`, `true`, or `yes` |
| `DJANGO_ALLOWED_HOSTS` | `*` | Comma-separated accepted hostnames |
| `DJANGO_SECURE_HSTS_SECONDS` | `31536000` | HSTS lifetime when debug mode is disabled |
| `DJANGO_PASSWORD_RESET_TIMEOUT` | `3600` | Password-reset token lifetime in seconds |
| `AXES_FAILURE_LIMIT` | `5` | Failed login attempts before lockout |
| `AXES_COOLOFF_TIME` | `1` | Axes lockout cool-off duration in hours |
| `DJANGO_EMAIL_BACKEND` | Console in debug; SMTP otherwise | Django email backend import path |
| `DJANGO_EMAIL_HOST` | `localhost` | SMTP server hostname |
| `DJANGO_EMAIL_PORT` | `25` | SMTP server port |
| `DJANGO_EMAIL_HOST_USER` | Empty | SMTP username |
| `DJANGO_EMAIL_HOST_PASSWORD` | Empty | SMTP password |
| `DJANGO_EMAIL_USE_TLS` | `false` | Enable SMTP STARTTLS |
| `DJANGO_EMAIL_USE_SSL` | `false` | Enable SMTP-over-SSL |
| `DJANGO_DEFAULT_FROM_EMAIL` | `no-reply@yamlconfig.local` | Sender used for account messages |

Boolean email settings recognize `1`, `true`, or `yes` as true.

Do not use the default secret key, wildcard hosts, or debug mode in production.

## Accounts and email

Account handling is provided by django-allauth.

Implemented account flows:

- Signup with username, email, and password confirmation
- Mandatory email verification
- Login using either username or email address
- POST-only logout
- Email-address management
- Password changes for authenticated users
- Password reset by emailed link
- Automatic login after signup email confirmation when the link is confirmed in
  the original signup browser session

The account templates live in `templates/account/` and extend the same base
layout used by the generator dashboard. Styling is applied to allauth's own
maintained forms through `generator/account_forms.py`.

In development, account emails are written to the server console. In
production, configure a working SMTP backend with the `DJANGO_EMAIL_*`
variables. Without working email delivery, new users cannot complete mandatory
verification or password resets.

django-axes protects password authentication. The default configuration locks
on the combination of username and IP address after five failures and resets
the failure count after a successful login.

## Using the web application

### Create a project

1. Sign in and select **New Project**.
2. Enter a project name.
3. Choose the primary preview target:
   - Docker Compose
   - Dockerfile
4. Save the project.

The target controls the content shown in the live preview. Both formats remain
available from the download buttons regardless of the selected preview target.

### Add services

A service can define:

- Name
- Image
- Build context
- Container name
- Startup command
- Restart policy
- Published ports
- Volume mounts
- Environment variables
- Dependencies on other project services
- An advanced `extra` object

Service names are unique within a project. Dependencies must reference services
that already exist in the same project and a service cannot depend on itself.

### Add networks

A project network can define:

- Name
- Driver
- External, attachable, and internal flags
- Labels
- Driver options
- An advanced `extra` object

Network names are unique within a project.

### Add named volumes

A named volume can define:

- Name
- Driver
- External flag
- Labels
- Driver options
- An advanced `extra` object

Volume names are unique within a project.

### Add scoped options

Every option has:

- A scope: `docker-compose` or `dockerfile`
- A key
- A value

Compose options are added to the top level of the generated Compose document.
Dockerfile options are only visible to the Dockerfile builder. The same key may
exist once in each scope, but cannot be duplicated inside one scope.

The form prevents known Dockerfile keys from being accidentally assigned to the
Compose scope and rejects unknown Dockerfile keys that the builder would ignore.

### Preview, copy, and download

The project detail page contains a read-only CodeMirror preview generated from
the current records. Available actions are:

- Copy the selected output
- Download a Docker Compose file
- Download a Dockerfile
- Download a ZIP containing both files

Downloaded filenames use a slug derived from the project name. The ZIP contains:

```text
docker-compose.yml
Dockerfile
```

## Web form input formats

The forms use readable line-based input while retaining compatibility with
existing JSON values.

### Lists

Ports, volume mounts, and service dependencies accept one value per line:

```text
8000:80
8443:443
```

The equivalent JSON list is also accepted:

```json
["8000:80", "8443:443"]
```

Duplicate list entries are removed while preserving their first occurrence.

### Key/value mappings

Environment variables, labels, and driver options accept one `KEY=value` pair
per line:

```text
APP_ENV=development
DEBUG=true
TOKEN=value=containing=equals
```

Only the first equals sign separates the key from the value. Empty values are
allowed:

```text
OPTIONAL_VALUE=
```

JSON objects remain supported:

```json
{
  "APP_ENV": "development",
  "DEBUG": true
}
```

### Advanced `extra` settings

The service, network, and named-volume `extra` fields require a JSON object:

```json
{
  "healthcheck": {
    "test": ["CMD", "curl", "-f", "http://localhost/health"],
    "interval": "30s"
  }
}
```

The object is merged into the corresponding generated block after normal
fields. An `extra` key can therefore override a generated key. This is
intentional flexibility, but malformed or contradictory settings can produce a
configuration Docker rejects.

## Data model

### `ConfigProject`

The project root contains:

- Owner
- Name
- Primary target type
- Creation timestamp
- Update timestamp

Deleting a user deletes their projects. Deleting a project cascades to its
services, options, networks, and named volumes.

### `Service`

Services belong to one project and are ordered by name. JSON database fields
store ports, volume mounts, environment variables, dependencies, and advanced
settings.

### `ProjectOption`

Options belong to one project and are separated by scope. A database constraint
enforces uniqueness for `(project, scope, key)`.

### `Network`

Networks belong to one project and have unique names inside that project.

### `NamedVolume`

Named volumes belong to one project and have unique names inside that project.

SQLite is the configured development database. `db.sqlite3` is ignored by Git.

## Generated output

### Docker Compose

`build_compose_yaml()` starts with:

```yaml
version: "3.8"
services: {}
```

For database-backed projects, the current Compose version is the builder default
because `ConfigProject` does not yet expose a version field in the web editor.

Supported service mappings:

| Project field | Compose key |
| --- | --- |
| `image` | `image` |
| `build_context` | `build` |
| `container_name` | `container_name` |
| `command` | `command` |
| `restart_policy` | `restart` |
| `ports` | `ports` |
| `volumes` | `volumes` |
| `environment` | `environment` |
| `depends_on` | `depends_on` |

Empty service values are omitted. Service `extra` settings are merged last.

Networks support driver, external, attachable, internal, labels, driver
options, and extra settings. Named volumes support driver, external, labels,
driver options, and extra settings.

Only options scoped to `docker-compose` are added to the Compose document.

### Dockerfile

The Dockerfile builder emits a basic sequence:

```dockerfile
FROM python:3.12-slim
WORKDIR /app
COPY . .
```

It can then append `RUN`, `EXPOSE`, and `CMD` instructions.

Supported Dockerfile option keys:

| Preferred key | Alias | Meaning |
| --- | --- | --- |
| `dockerfile_service` | `primary_service` | Service used for image, port, and command defaults |
| `dockerfile_from` | `base_image` | Base image for `FROM` |
| `dockerfile_workdir` | `workdir` | Working directory |
| `dockerfile_copy` | — | Arguments for `COPY` |
| `dockerfile_run` | `run` | Build command or commands |
| `dockerfile_cmd` | `cmd` | Default container command |
| `dockerfile_expose` | — | Container ports to expose |

Selection and fallback behavior:

1. If `dockerfile_service` or `primary_service` names an existing service, that
   service supplies defaults.
2. Otherwise, the first service with a non-empty image is selected.
3. If no usable image exists, `FROM` falls back to `python:3.12-slim`.
4. `WORKDIR` defaults to `/app`.
5. `COPY` defaults to `. .`.
6. Ports default to the selected service's container-side ports.
7. The selected service's command becomes `CMD` unless an option overrides it.

Dockerfile hardening behavior:

- Blank or multiline values cannot create an empty or injected `FROM`,
  `WORKDIR`, or `COPY` instruction.
- Each non-empty line in `dockerfile_run` becomes a separate `RUN`.
- Host port mappings are reduced to their container port for `EXPOSE`.
- Exposed ports must be between 1 and 65535.
- `/tcp` and `/udp` are accepted and normalized to lowercase.
- Duplicate ports are removed.
- String commands are JSON-encoded as `["sh", "-c", "..."]`.
- JSON-list commands are emitted in exec form.

`dockerfile_expose` accepts comma-separated values, whitespace-separated values,
or a JSON list:

```text
8000, 8443/udp
```

```json
["8000", "8443/udp"]
```

## Python builder API

The generator is independent of Django views and lives in
`generator/yaml_builder.py`.

Public entry points:

```python
build_compose_yaml(project)
build_dockerfile(project)
build_output(project, target_type=None)
build_yaml(project)  # backward-compatible Compose alias
```

`project` can be:

- A dictionary
- An object with matching attributes
- A Django `ConfigProject` instance with related managers

### Dictionary input

Required project data:

- `services`: iterable of service dictionaries or objects

Recommended project data:

- `target_type`: `docker-compose` or `dockerfile`
- `version`: Compose version; defaults to `3.8`
- `networks`
- `named_volumes` or the legacy alias `volumes`
- `options`

Example:

```python
from generator.yaml_builder import build_output

project = {
    "target_type": "docker-compose",
    "version": "3.8",
    "services": [
        {
            "name": "web",
            "image": "nginx:alpine",
            "ports": ["8080:80"],
            "environment": {"APP_ENV": "development"},
            "depends_on": ["database"],
        },
        {
            "name": "database",
            "image": "postgres:16",
            "environment": {"POSTGRES_PASSWORD": "change-me"},
            "volumes": ["postgres-data:/var/lib/postgresql/data"],
        },
    ],
    "networks": [
        {
            "name": "application",
            "driver": "bridge",
        }
    ],
    "named_volumes": [
        {
            "name": "postgres-data",
            "driver": "local",
        }
    ],
    "options": [
        {
            "scope": "docker-compose",
            "key": "name",
            "value": "example-project",
        },
        {
            "scope": "dockerfile",
            "key": "dockerfile_from",
            "value": "python:3.12-slim",
        },
        {
            "scope": "dockerfile",
            "key": "dockerfile_run",
            "value": "pip install -r requirements.txt",
        },
        {
            "scope": "dockerfile",
            "key": "dockerfile_cmd",
            "value": "python app.py",
        },
    ],
}

selected_output = build_output(project)
compose_output = build_output(project, target_type="docker-compose")
dockerfile_output = build_output(project, target_type="dockerfile")
```

For backward compatibility, dictionary options without `scope` are inferred:

- Known Dockerfile keys use the `dockerfile` scope.
- All other keys use the `docker-compose` scope.
- An invalid explicit scope is ignored rather than leaking into either output.

Services without a name are skipped during Compose generation.

## Routes and administration

Main route groups:

| URL | Purpose |
| --- | --- |
| `/` | Current user's project list |
| `/projects/create/` | Create a project |
| `/projects/<id>/` | Project detail and live preview |
| `/projects/<id>/download/compose/` | Download Compose YAML |
| `/projects/<id>/download/dockerfile/` | Download Dockerfile |
| `/projects/<id>/download/bundle/` | Download both files as ZIP |
| `/accounts/` | django-allauth account routes |
| `/admin/` | Django administration |

Each project also exposes nested create, edit, and delete routes for services,
options, networks, and named volumes. Mutating operations use POST for the final
change and include Django CSRF protection.

The admin registers all generator models and supports filtering and searching.
Admin or direct ORM changes affect generated output immediately because output
is generated live.

## Production considerations

The repository is configured for convenient local development, not for
immediate public deployment.

Before production:

1. Set a strong, private `DJANGO_SECRET_KEY`.
2. Set `DJANGO_DEBUG=false`.
3. Replace wildcard hosts with explicit `DJANGO_ALLOWED_HOSTS`.
4. Configure a real SMTP provider and verify account-email delivery.
5. Serve the application behind HTTPS.
6. Ensure the reverse proxy sends `X-Forwarded-Proto: https`; the settings trust
   that header through `SECURE_PROXY_SSL_HEADER`.
7. Review HSTS before enabling preload for a real domain.
8. Run `python manage.py collectstatic`.
9. Use a production-capable WSGI or ASGI server.
10. Consider replacing SQLite with a production database.
11. Back up the database and protect all environment secrets.
12. Replace or locally bundle CDN frontend dependencies if the deployment must
    work without third-party asset availability.
13. Run tests and `python manage.py check --deploy` against the production
    environment.

When debug mode is disabled, the project enables:

- Secure session cookies
- Secure CSRF cookies
- HTTPS redirects
- HSTS
- HSTS subdomain inclusion
- HSTS preload signaling

These settings assume HTTPS is configured correctly. Test them in a staging
environment before public deployment.

## Current limitations

- Only Docker Compose and Dockerfile generation are implemented.
- There is no REST or GraphQL API.
- There is no configuration import feature.
- Generated YAML is preview-only; raw text is not directly editable.
- The web editor does not expose the Compose version, so it uses `3.8`.
- Compose-scoped options are flexible top-level keys; users can still create
  combinations unsupported by Docker Compose.
- Generated files are not passed through Docker for semantic validation.
- Advanced `extra` objects can override generated values.
- Dockerfile generation produces one Dockerfile from a selected primary service,
  not one Dockerfile per service.
- SQLite is the only configured database backend.
- Frontend libraries are loaded from public CDNs.
- The `theme/` directory is scaffolding and is not an active asset pipeline.
- No production container image, deployment manifest, or CI workflow is
  included yet.

Potential future work:

- Ansible playbook generation
- GitHub Actions workflow generation
- Kubernetes manifest generation
- Configuration import and round-trip editing
- Versioned project revisions
- A documented API
- Local frontend asset builds
- Production deployment examples

## Troubleshooting

### `ModuleNotFoundError` for Django, axes, or allauth

Activate the virtual environment and install the pinned requirements:

```bash
source venv/bin/activate
python -m pip install -r requirements.txt
```

### A new account cannot sign in

Email verification is mandatory. Open the verification link sent by email. In
development, inspect the terminal running `manage.py runserver`.

### Verification or reset emails are not arriving

- In debug mode, the default backend prints messages to the console.
- In production, verify every `DJANGO_EMAIL_*` value.
- Do not enable both TLS and SSL unless the SMTP provider explicitly requires
  that combination.
- Confirm that `DJANGO_DEFAULT_FROM_EMAIL` is accepted by the provider.

### Login is locked after repeated failures

django-axes locks authentication after the configured failure limit. Wait for
the configured cool-off period or use the axes management/admin tools to reset
the relevant failure records.

### CSS or the YAML preview is missing

The browser must currently reach the Tailwind, Alpine.js, and CodeMirror CDNs.
Check the browser network panel, content-security policy, firewall, and internet
connection.

### Migrations are missing

```bash
python manage.py migrate
python manage.py showmigrations
```

### The preview differs from expected configuration

- Confirm the project's selected target.
- Check each option's scope.
- Check whether an `extra` object overrides a normal field.
- For Dockerfiles, verify which service is selected by `dockerfile_service`.
- Remember that blank optional values are omitted.
